The Competitive Analysis

An evidence-based assessment of Eiffel + AI development, challenging conventional assumptions.

December 2025 | Larry Rix and Claude (Anthropic) | Version 2.0

Executive Summary

This analysis presents findings from 13 days of AI-assisted Eiffel development, producing 25 libraries + 4 applications with 1,200+ tests. The evidence challenges several conventional assumptions about language ecosystem advantages.

Key findings:

  • Library ecosystem: 25 libraries built in 13 days; creation velocity makes building viable
  • Tooling requirements: AI-assisted workflows reduce IDE dependency
  • Quality assurance: Design by Contract catches AI-generated errors at runtime

This wasn't an anomaly. It demonstrates the achievable velocity when AI assistance is combined with proper reference documentation and workflow patterns.

Addressing Developer Availability

The conventional concern: "You can't hire Eiffel developers."

The evidence suggests a different framing:

  • Training track record: 12+ developers trained over 5 years, including complete beginners
  • Training duration: 5 days to productivity for OOP developers
  • Simplicity advantage: Eiffel's consistent syntax and single paradigm (DBC) reduce learning curve
  • AI acceleration: Reference documentation + AI provides instant pattern guidance

The constraint is trainer capacity (5 days per developer), not talent availability. This is a solvable resource allocation problem.

  • Ownership advantage: You control updates, understand the implementation, fix issues directly

Design by Contract addresses this systematically:

  • Preconditions validate all inputs before execution
  • Postconditions verify outputs match specifications
  • Invariants ensure consistent object state
  • Runtime verification catches errors at first execution, not production

From Bertrand Meyer's "AI for Software Engineering: From Probable to Provable" (CACM 2025): AI produces statistically likely code, not proven correct code. DBC provides the verification layer.

Acknowledged Limitations

An honest assessment requires acknowledging genuine challenges:

  • IDE ecosystem: EiffelStudio is the primary IDE; no VS Code LSP support
  • Community size: Smaller community means fewer Stack Overflow answers and tutorials
  • Commercial perception: May require justification in enterprise contexts

These are real constraints, but they're mitigated by AI-assisted workflows and reference documentation systems.

Implications

For technical decision makers evaluating language choices:

  • Developer availability is a training problem, not a hiring problem
  • Library availability is a velocity problem, not an ecosystem problem
  • Runtime verification provides unique value for AI-generated code
  • Productivity multipliers of 40-80x change the economics significantly
